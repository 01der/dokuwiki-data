===== What's the Point to create a SMIL Player for Digital Sigature ? =====
  *What's the point to create a SMIL player for digital sigature ?
    -Check all video players.
      -Video codecs.
        -Can it play MPEG2 & MPEG4 videos ? Is it shipped with its own video codecs, or does it use system video codecs ?
          -Linux: May it make use of video codecs by "Dependancies" ? 
          -Windows: Is the player shipped with its own codecs, or does it make use of the codecs installed in the Windows operating system ? In the latter case, you might have to install a DVD player which allows to use its MPEG2 codec externally by third party applications.
        -Can it play the MPEG2 & MPEG4 videos indeed that your customers supplied as samples ? Have a test.. 
      -API.
        -Source code available ? If so, in which programming languages is the sourcecode written 
        -Does the player have a application programming interface ( API ) ? 
        -Which programming languages are supported by the video player and optionally by the browser ( remember Apple disabled the Java interface for Quicktime, on MacOSX... ) ?
      -Player behaviour.
        -Can the player be started and started to play at start of operating system ? Windows: Does it start properly if it is stored / linked in the startup folder ?
        -If there is no network access at start, may it play the last playlist stored on the device ? Alternatively, can it wait for network access ? -> loose coupling.
        -If there is no network access at startup, but some time later later, can it be instructed to load a new playlist from network, then ? -> loose coupling.
        -Time scheduling: Playing a contents for X minutes, playing contents at a certain time, for a certain time of day / week / month / year?
        -Does the player support configuration and start commands by "command line" commands ? Are there sufficient "command line" commands for automatic operation, for the dedicated purposes ? E.g. "Get a playlist from an URL at start and play it".
        -How to start the player in fullscreen mode ? 
        -If it is a video player, can it play other contents ( images, text ) in its player window, too ? 
        -Can you assign a unique user ID to a dedicated player which is passsed as [[http://en.wikipedia.org/wiki/User_agent|User Agent ID]] when accessing a digital signage contents server ?
        -Can the player share its window with other contents, like in a browser window ? 
        -Does it support multi-layer dislays, i.e. can a graphics or text cover a video running in the background, and vice versa ?
  *Creation.
    -Concentrate on running videos. So running a video ist the first milestone / quality gate, nothing which is kept in the product backlog for the end of the project or even after the project is closed.
    -Create a minimalistic SMIL interface, in a programming language supported by the API of the player so that the device can play videos in the same player window as all other SMIL media ( graphics / pictures / photos, text ) is played. 
    -Xtreme programming approach is welcome ( estimte what features cost you what time for implementation,.. ask your customer about priorities of features,.. ).
    -Do the rest that the customer wants...
  *Reasons.
    -Nobody wants to create contents in a contents management system, nor in a player. Contents is created by professionals using their professional tools, and then exported as video or graphics.
    -The customers does not just wants to play videos, but the customer also wants to have "animated still pictures", as this makes a slideshow much more interesting for the viewer ( the customer of the customer, at the point of sale where digital signage is used ). 
    -The customer wants the solution very quicky, and does not want to pay for the implementation of features not used in the show at the point of sale, finally.
